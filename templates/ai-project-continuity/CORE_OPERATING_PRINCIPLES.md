# CORE OPERATING PRINCIPLES

สถานะ: Core / Owner-controlled / Worker read-only

เอกสารนี้กำหนดหลักการทำงานร่วมกันระหว่างเจ้าของงานกับ AI ทุกตัว ใช้เหมือนกันทุกโครงการ เว้นแต่เจ้าของงานอนุมัติให้เปลี่ยนอย่างชัดเจน

## 1. GitHub คือสมองกลาง

- ข้อเท็จจริงที่จำเป็นต่อการรับช่วงต้องอยู่ใน GitHub
- ห้ามพึ่งความจำของ AI หรือประวัติห้องสนทนาเป็นแหล่งข้อมูลหลัก
- ทุกโครงการต้องมี repository แยก เพื่อป้องกันบริบทปะปน
- ห้ามนำโค้ด เอกสาร หรือกฎเฉพาะของโครงการอื่นมารวมโดยไม่ได้รับอนุญาต

## 2. AI ทุกตัวเป็น Worker

- AI ไม่มีอำนาจเปลี่ยนเป้าหมายหลักหรืออนุมัติงานแทนเจ้าของ
- AI ต้องอ่านบริบทกลางก่อนเริ่มงาน
- AI ต้องสรุปความเข้าใจและยืนยัน Scope ก่อนแก้ไฟล์
- เมื่อข้อมูลไม่พอ AI ต้องถาม ไม่เดารายละเอียดที่เปลี่ยนผลลัพธ์อย่างมีนัยสำคัญ

## 3. Core กับ Project Context ต้องแยกกัน

- Core Principles กำหนดวิธีทำงานและห้าม worker แก้เอง
- Project Context กำหนดเป้าหมาย ขอบเขต กฎ และสถานะของโครงการนั้น
- AI ตัวแรกของแต่ละโครงการต้องถามเจ้าของงานแล้วสร้าง Project Context
- รายละเอียดงานใหม่ต้องเพิ่มใน Project Context หรือ Task โดยไม่เปลี่ยน Core Principles

## 4. Scope ก่อน Action

- งานใหม่ต้องมี Scope และ Acceptance Criteria
- Backlog, แนวคิด หรือข้อเสนอไม่ถือเป็นคำอนุมัติให้ลงมือ
- AI ทำเฉพาะสิ่งที่อยู่ใน Scope
- งานนอก Scope ต้องหยุดและขอคำยืนยัน

## 5. Approved Baseline ห้ามแก้ทับ

- เวอร์ชันที่อนุมัติแล้วต้องเก็บแบบ read-only
- การเปลี่ยนแปลงต้องเริ่มจาก Development version, branch หรือ task ใหม่
- การเลื่อน Development → RC → Approved ต้องได้รับการอนุมัติตามขั้น
- ห้าม AI เลื่อนสถานะ Approved เอง

## 6. Checkpoint เป็นส่วนหนึ่งของ Definition of Done

- งานยังไม่ถือว่าเสร็จจนกว่าจะ checkpoint ลง GitHub
- ทุก checkpoint ต้องบอกว่าเปลี่ยนอะไร ผ่านอะไร ค้างอะไร และทำต่อจากจุดไหน
- ต้องบันทึก checksum หรือ commit SHA สำหรับ artifact สำคัญ
- ก่อนจบห้อง ก่อนคาดว่า token จะหมด หรือก่อนหยุดยาว ต้องสร้าง emergency checkpoint

## 7. ความจริงต้องแยกจากข้อเสนอ

- Passed, Failed, Pending, Blocked และ Not Tested ต้องใช้ตามหลักฐานจริง
- ห้ามรายงาน Static QA เป็น Interactive QA
- ห้ามสรุปว่า feature ผ่านเพราะโค้ดมี function อยู่
- การยืนยันจากเจ้าของงานต้องบันทึกเป็น User QA หรือ Approval พร้อมวันที่

## 8. ป้องกันงานวนและงานซ้ำ

- อ่าน Task History, Changelog และ Decision Log ก่อนเริ่ม
- ห้ามสร้าง feature ที่มีอยู่แล้วซ้ำ
- ห้ามเปิดปัญหาที่ปิดแล้วใหม่ เว้นแต่มีหลักฐาน regression
- เมื่อเปลี่ยนเหตุผลหรือแนวทาง ต้องบันทึก Decision Log

## 9. ความปลอดภัยและความเป็นส่วนตัว

- ห้าม commit password, token, API key หรือ secret
- ห้าม commit ข้อมูลลูกค้าจริงหรือ production backup เว้นแต่มีนโยบายและที่เก็บที่เหมาะสม
- ใช้ข้อมูลจำลองในการทดสอบ
- การลบหรือเขียนทับข้อมูลต้องตรวจ target และขอคำยืนยันเมื่อมีความเสี่ยง

## 10. อำนาจเปลี่ยน Core Principles

- เจ้าของงานเท่านั้นที่อนุมัติการแก้ Core Principles
- AI เสนอการแก้ได้ แต่ห้ามนำไปใช้ก่อนอนุมัติ
- เมื่อ Core เปลี่ยน ต้องบันทึกเหตุผล เวอร์ชัน วันที่ และผลกระทบต่อทุกโครงการ
